The problem

Capturing memory is easy. Trusting it is the hard part.

Most agent-memory tools optimise for capture: every session writes down what it learned. The failure mode is built in — a confident agent recites something it inferred once and got wrong, and the next agent inherits the mistake. loreguard inverts the primitive. It optimises for trust after capture.

How it works

Agents suggest. Humans approve.

One closed loop turns scattered session knowledge into a single trusted record per topic.

MCP surface

Five tools your agent already knows when to call

Each tool description leads with when to call it, so retrieval becomes an ambient habit even in agents with zero loreguard awareness.

search_lore

Find ratified positions

Compact summaries ranked by relevance. On a miss it coaches the agent on what to do next instead of dead-ending.

get_lore

Full record on demand

Pull the complete body only when needed. Restricted records are env-gated, so sensitive lore never leaks by default.

suggest_lore

Draft, never publish

Agent-suggested records always enter the review queue as drafts. Duplicate hints and structured errors keep them clean.

report_conflict

Flag code-vs-lore drift

When the codebase disagrees with a record, the agent files a draft counter-record. The original is never mutated.

record_absence

Mark a known gap

"We checked — no policy here." Self-expiring markers stop agents re-searching the same void. Off by default via MCP.

Trust model

Guardrails that hold under pressure

The trust primitives aren't bolted on — they're enforced at every boundary and covered by tests.

  • Agent writes are draftsNothing an agent suggests becomes trusted until a human approves it.
  • Confidence is clampedDrafts can't claim high confidence; records without a source can't either.
  • Restricted records are gatedExcluded from default search and env-gated for direct fetch over MCP.
  • Sanitised audit logSensitive fields are recorded as character counts only — never raw text.
  • Local-first, privateOne SQLite file at ~/.loreguard/lore.db, mode 0600. No cloud.
  • PR-reviewable syncExport records to Markdown so changes ride through normal code review.

FAQ

Questions, answered

How is this different from agent memory?

Generic memory stores what one session inferred. loreguard stores what a team has reviewed and approved. Agents can suggest records, but a record only becomes trusted lore after a human approves it — so the team gets one ratified record per topic instead of N parallel, possibly-wrong beliefs.

Why not just use CLAUDE.md?

CLAUDE.md is always-on context that every prompt pays for. loreguard is just-in-time: the agent calls search_lore only when a task warrants it and gets a compact summary of what's already decided. They're complementary — many teams keep a short CLAUDE.md retrieval rule that points agents at loreguard.

Where does the data live? Is anything sent to the cloud?

Everything is local. Records live in a SQLite database at ~/.loreguard/lore.db (file mode 0600). There's no cloud component, and read-tracking is off by default. You can export records to Markdown for PR-reviewable sync across a team.

Which agents and editors does it support?

Any client that speaks the Model Context Protocol (MCP) — including Claude Code and Cursor. loreguard ships an MCP server for agents plus a human-facing CLI for review, approval, and governance.

What does it cost?

Nothing. loreguard is free and open source under the MIT license.
